Ebuild 快速入门指南

此页面非常简明地介绍了如何编写 ebuild。 它不会试图涵盖大量的细节或者只有开发人员才会接触到的问题， 而是给出一些有助于掌握 ebuild 工作原理的基本思想的小例子。

有关所有输入输出的全面介绍，请阅读 Ebuild Writing 章节。 General Concepts 章节也很有用。

需要说明的是本文中用到的例子，都在真正的 ebuild 树的基础上，对若干个部分做了阉割、修改与简化。

第一个 Ebuild

我们从源码索引工具 Exuberant Ctags 的 ebuild 开始入门。 下面是一个简化的 dev-util/ctags/ctags-5.5.4.ebuild（你可以在主树中查看真正的 ebuild）。 

# Copyright 1999-2021 Gentoo Authors
# Distributed under the terms of the GNU General Public License v2

EAPI=6

DESCRIPTION="Exuberant ctags generates tags files for quick source navigation"
HOMEPAGE="https://ctags.io/ https://github.com/universal-ctags/ctags"
SRC_URI="mirror://sourceforge/ctags/${P}.tar.gz"

LICENSE="GPL-2+"
SLOT="0"
KEYWORDS="~mips ~sparc ~x86"

src_configure() {
	econf --with-posix-regex
}

src_install() {
	emake DESTDIR="${D}" install

	dodoc FAQ NEWS README
}

基本格式

如你所见，ebuild 只是一个在特殊环境下执行的 bash 脚本。

ebuild 最顶上那2行是 ebuild 头。在任何 ebuild 中都会出现。

ebuild 使用制表符缩进，每个制表符代表4个空格。参阅 Ebuild File Format。

变量

接下来是一系列变量。这些变量告诉了 Portage 关于 ebuild 和所提及软件包的各种信息。

EAPI： ebuild 的 EAPI，参阅 EAPI Usage and Description。

DESCRIPTION： 对软件包及其用途的简短描述。

HOMEPAGE： 软件包主页的链接（记得包含 URI 协议名，例如 https://）。

SRC_URI： 告诉 Portage 用来下载源码的链接。 在此处，mirror://sourceforge/ 是一种特殊的记法，意为“任何一个 Sourceforge 镜像站”。 ${P} 是一个由 Portage 赋值的只读变量，其值为软件包的名称与版本。 在这个例子中，它将是 ctags-5.5.4。

LICENSE： 许可证，GPL-2+（GNU通用公共许可证第2版或更新的版本）。

SLOT： 此变量告诉 Portage 应该把软件包安装到哪个槽里。 如果你以前没见过槽，就写"0"或参阅 Slotting。

KEYWORDS： 此变量是 ebuild 在上面测试过的架构。 对于最近编写的 ebuild 一般使用~关键字来表示软件包没有直接提交稳定版， 即使它看上去工作正常。 更多细节请参阅 Keywording and Stabilization。

构建函数

接下来是一个叫做 src_configure 的函数。 Portage 会在配置软件包的时候调用它。 econf 函数是 ./configure 命令的封装。 如果出于某种原因 econf 报错了，Portage 将会停下而不是尝试继续安装。

src_install 函数在 Portage 安装 软件包的时候调用。 这里有一点点微妙 —— 我们需要安装到由 ${D} 变量（由 Portage 赋值 —— 参阅 Install Destinations 和 Sandbox）指定的特殊路径下，而不是直接安装到当前的文件系统中。

附注： 规范的安装方法是 emake DESTDIR="${D}" install。 这用在任何正确编写的标准 Makefile 上都正常。 如果这导致沙箱报错，请阅读 src_install 来获知如何手动安装。

dodoc 是一个辅助函数，用来安装 /usr/share/doc 目录下的相关文件。

Ebuild 可以定义其他的函数（参阅 Ebuild Phase Functions）。 在所有的情况下，Portage 提供了合理的、通常能做‘正确事情’的默认实现。 比如本例就无需定义 src_unpack 和 src_compile 函数， src_unpack 函数用来解压源码和打补丁，默认的实现就可以完成本例中所需的一切操作。 类似的，src_compile 函数会调用 make 的封装 emake。

附注： 以前为了检查错误，在每一条命令后边都要加上 || die。 从 EAPI 4 开始就没必要了，Portage 提供的函数会在某步操作失败后自动终止。

带有依赖的 Ebuild

在 ctags 的例子中，我们没有告诉 Portage 它的任何依赖。巧的是，这么做没有问题， 因为编译和运行 ctags 仅仅只需要基本的工具链（至于为什么不需要显式指定这些依赖， 请参阅 Implicit System Dependency）。 然而，人生不如意事十之八九。

这是 app-misc/detox/detox-1.1.1.ebuild： 

# Copyright 1999-2021 Gentoo Authors
# Distributed under the terms of the GNU General Public License v2

EAPI=5

DESCRIPTION="detox safely removes spaces and strange characters from filenames"
HOMEPAGE="http://detox.sourceforge.net/"
SRC_URI="mirror://sourceforge/${PN}/${P}.tar.bz2"

LICENSE="BSD"
SLOT="0"
KEYWORDS="~hppa ~mips sparc x86"

RDEPEND="dev-libs/popt"
DEPEND="${RDEPEND}
	sys-devel/flex
	sys-devel/bison"

src_configure() {
	econf --with-popt
}

又一次，你可以看到 ebuild 头和一些变量。 在 SRC_URI变量中， ${PN} 表示不带版本后缀的软件包名称（这类变量的更多介绍可以参阅 Predefined Read-Only Variables）。

我们只定义了 src_configure 函数。 src_install 函数不需要定义， 因为对于本软件包来说调用 emake install 并安装文档的 src_install 默认实现可以正确生效。

DEPEND 和 RDEPEND 分别是软件包的编译时依赖和运行时依赖。 查看 Dependencies 来获知更多复杂的例子。

带有补丁的 Ebuild

通常我们需要给软件包打补丁。这是在 src_prepare 函数中用 eapply 附注函数完成的。 只有在 EAPI 7 中才能使用 eapply。 下面是 app-misc/detox/detox-1.1.0.ebuild： 

# Copyright 1999-2021 Gentoo Authors
# Distributed under the terms of the GNU General Public License v2

EAPI=7

DESCRIPTION="detox safely removes spaces and strange characters from filenames"
HOMEPAGE="http://detox.sourceforge.net/"
SRC_URI="mirror://sourceforge/${PN}/${P}.tar.bz2"

LICENSE="BSD"
SLOT="0"
KEYWORDS="~hppa ~mips ~sparc ~x86"

RDEPEND="dev-libs/popt"
DEPEND="${RDEPEND}
	sys-devel/flex
	sys-devel/bison"

src_prepare() {
	eapply "${FILESDIR}"/${P}-destdir.patch \
		"${FILESDIR}"/${P}-parallel_build.patch
	eapply_user
}

src_configure() {
	econf --with-popt
}

注意 ${FILESDIR}/${P}-destdir.patch 指的是 detox-1.1.0-destdir.patch，它存放在 Gentoo 仓库的 files/ 子目录中。更大的补丁文件需要放到 dev.gentoo.org 上你的开发空间中而不是 files/ 目录或镜像站中 —— 参阅 Gentoo Mirrors 和 Patching with epatch and eapply。

当你重写了 src_prepare， 请确保在函数中调用了 eapply_user。

带有 USE 标志的 Ebuild

现在我们讨论 USE 标志。下面是 dev-libs/libiconv/libiconv-1.9.2.ebuild， 一个 libc 没有实现的字符集转换库。 

# Copyright 1999-2021 Gentoo Authors
# Distributed under the terms of the GNU General Public License v2

EAPI=5

DESCRIPTION="GNU charset conversion library for libc which doesn't implement it"
HOMEPAGE="https://www.gnu.org/software/libiconv/"
SRC_URI="ftp://ftp.gnu.org/pub/gnu/libiconv/${P}.tar.gz"

LICENSE="LGPL-2+ GPL-3+"
SLOT="0"
KEYWORDS="~amd64 ~ppc ~sparc ~x86"
IUSE="nls"

DEPEND="!sys-libs/glibc"

src_configure() {
	econf $(use_enable nls)
}

注意 IUSE 变量。 它列出了所有（非特殊的）ebuild 用到的 use 标志。这用在 emerge -pv 的输出，以及一些其他东西中。

软件包的 ./configure 脚本通常会读取 --enable-nls 或 --disable-nls 参数。我们使用 use_enable 工具函数自动生成它， 生成哪个取决于用户配置的 USE 标志（参阅 Query Functions Reference）。

另一个更复杂的示例，这次基于 mail-client/sylpheed/sylpheed-1.0.4.ebuild： 

# Copyright 1999-2021 Gentoo Authors
# Distributed under the terms of the GNU General Public License v2

EAPI=7

inherit desktop

DESCRIPTION="A lightweight email client and newsreader"
HOMEPAGE="https://sylpheed.good-day.net/"
SRC_URI="mirror://sourceforge/${PN}/${P}.tar.bz2"

LICENSE="GPL-2+ LGPL-2.1+"
SLOT="0"
KEYWORDS="alpha amd64 hppa ia64 ppc ppc64 sparc x86"
IUSE="crypt imlib ipv6 ldap nls pda ssl xface"

RDEPEND="=x11-libs/gtk+-2*
	crypt? ( >=app-crypt/gpgme-0.4.5 )
	imlib? ( media-libs/imlib2 )
	ldap? ( >=net-nds/openldap-2.0.11 )
	pda? ( app-pda/jpilot )
	ssl? ( dev-libs/openssl )
	xface? ( >=media-libs/compface-1.4 )
	app-misc/mime-types
	x11-misc/shared-mime-info"
DEPEND="${RDEPEND}
	dev-util/pkgconfig
	nls? ( >=sys-devel/gettext-0.12.1 )"

src_prepare() {
	eapply "${FILESDIR}"/${PN}-namespace.diff \
		"${FILESDIR}"/${PN}-procmime.diff
	eapply_user
}

src_configure() {
	econf \
		$(use_enable nls) \
		$(use_enable ssl) \
		$(use_enable crypt gpgme) \
		$(use_enable pda jpilot) \
		$(use_enable ldap) \
		$(use_enable ipv6) \
		$(use_enable imlib) \
		$(use_enable xface compface)
}

src_install() {
	emake DESTDIR="${D}" install

	doicon sylpheed.png
	domenu sylpheed.desktop

	dodoc [A-Z][A-Z]* ChangeLog*
}

注意可选依赖。某几行的 use_enable 使用了2个参数的版本 —— 这在 USE 标志的名称不能精确匹配 ./configure 参数的时候非常有用。
